# Часть 1: Code Review

В папке [`workflows/`](workflows/) три n8n-воркфлоу. Каждый - упрощённая модель того, что мы реально делаем в проекте: регистрация юзера, опрос внешнего API по расписанию, аплоад файлов в S3 (MinIO).

В каждом я **специально** заложил по 2-4 проблемы. Где-то очевидные, где-то тонкие.

## Что делать

1. Подними окружение из [`../setup/`](../setup/), импортируй три JSON-файла в свой self-hosted n8n.
2. Прокликай каждый воркфлоу, посмотри ноды, expressions, parameters, settings.
3. Найди все проблемы. Приоритет проблем:
   - **high** - security, потеря данных.
   - **medium** - неправильное поведение в edge case, лишний расход внешних API, тихие сбои без алертов.
   - **low** - неудобно поддерживать, читать, дебажить.
4. Возьми **топ-3** из найденных и реализуй фиксы прямо в JSON, оставь нотисы на нодах, которые исправлял. Сохрани исправленные версии в [`workflows/fixed/`](workflows/) с тем же именем.
5. Заполни [`deliverable-template.md`](deliverable-template.md) - там форма отчёта.

## Воркфлоу

### `01-mini-register.json`

Регистрация юзера: webhook принимает email + password, валидирует, проверяет уникальность email, хеширует пароль, создаёт запись в `users`, шлёт verification-код на email через SendGrid.

Поднятый локально n8n + Postgres из `setup/` подойдут. SendGrid реально дёргать не надо - смотри код, не его реальный ответ.

### `02-job-poller.json`

Cron-воркфлоу. Раз в N минут берёт из `jobs` все записи в статусе `pending`, дёргает внешний API чтобы узнать их актуальный статус, обновляет в БД.

API дёргать не надо, это про логику.

### `03-file-uploader.json`

Webhook принимает файл в multipart/form-data, проверяет размер, заливает в MinIO, создаёт запись в `files`, отдаёт юзеру URL для скачивания.

MinIO у тебя локальный (из `setup/`), креды прописаны в воркфлоу через `S3 (local)` credential - тебе надо его создать в n8n с теми же ключами, что в docker-compose (`minioadmin / minioadmin`).

## На что смотреть

Подсказки по типажу проблем:
- Hardcoded ключи / токены / пароли.
- Settings ноды: `alwaysOutputData`, `continueOnFail`, retries, timeouts.
- Cron-расписания: что они реально означают.
- Expressions: правильно ли тянутся данные между нодами.
- Response status codes: что отвечаем при ошибке.
- Validation: проверяется ли вход на стороне n8n или мы доверяем клиенту.
- Polling: max attempts, backoff, timeout.
- Error handling: есть ли `errorWorkflow`, что произойдёт при сбое.
- Path injection: как формируются пути к файлам / SQL / shell-командам.

Это далеко не весь список, и не во всех воркфлоу есть проблемы из всех категорий.

## Формат сдачи

- Заполненный [`deliverable-template.md`](deliverable-template.md).
- Папка `workflows/fixed/` с тремя обновлёнными JSON.
- Если есть мысли - можно лучше, в отчёте есть отдельный блок.
